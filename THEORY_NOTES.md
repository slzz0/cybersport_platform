# Теоретический конспект по frontend, Docker и деплою

Этот файл собран как простой и понятный конспект по темам, которые связаны с проектом `GRACEIT`: frontend, SPA, адаптивность, UI/UX, Docker, CI/CD и хостинг.

## 1. "Толстый" и "тонкий" клиент

### Тонкий клиент
Тонкий клиент выполняет минимум логики на устройстве пользователя. Основная обработка данных происходит на сервере.

Признаки тонкого клиента:
- большая часть логики находится на backend
- клиент в основном отправляет запросы и показывает ответ
- меньше требований к устройству пользователя
- сильная зависимость от интернета и сервера

Плюсы:
- проще поддерживать бизнес-логику в одном месте
- легче контролировать безопасность
- меньше требований к слабым устройствам

Минусы:
- без сервера и сети такой клиент почти бесполезен
- больше нагрузка на backend

### Толстый клиент
Толстый клиент выполняет заметную часть логики на стороне пользователя.

Признаки толстого клиента:
- часть логики, состояния и интерфейсной обработки находится в браузере
- активно используется JavaScript/TypeScript
- интерфейс может обновляться без полной перезагрузки страницы

Плюсы:
- более быстрый и интерактивный интерфейс
- меньше перезагрузок страницы
- часть нагрузки переносится с сервера на клиент

Минусы:
- выше требования к браузеру и устройству
- сложнее frontend-часть
- могут появляться сложности с SEO

### Что используется в этом проекте
В проекте `GRACEIT` frontend на `React + TypeScript` является скорее примером толстого клиента, потому что:
- интерфейс работает как SPA
- логика отображения, фильтрации, состояний загрузки и ошибок находится на frontend
- backend в основном отдает данные через REST API

## 2. SPA

SPA (`Single Page Application`) — это одностраничное приложение, которое загружается в браузер один раз, а дальше обновляет контент без полной перезагрузки страницы.

Как это работает:
- браузер загружает основной HTML, CSS и JavaScript
- после этого переходы между страницами происходят внутри приложения
- данные подгружаются через API-запросы

Плюсы SPA:
- быстрые переходы между экранами
- плавный UX
- удобно строить сложные интерфейсы

Минусы SPA:
- первая загрузка может быть тяжелее
- есть сложности с SEO
- многое зависит от корректной работы JavaScript

### Что в проекте является SPA
`GRACEIT` — это SPA, потому что:
- используется `React Router`
- страницы меняются без полной перезагрузки
- данные подгружаются через API

## 3. Адаптивная верстка

Адаптивная верстка — это подход, при котором интерфейс корректно отображается на разных устройствах:
- телефонах
- планшетах
- ноутбуках
- больших мониторах

Для этого обычно используют:
- `flexbox`
- `grid`
- медиазапросы
- адаптивные отступы, размеры шрифтов и сетки

### Зачем это нужно
- пользователь может открыть приложение на любом устройстве
- интерфейс остается удобным
- проект выглядит профессиональнее

### Что в проекте
Во frontend используются адаптивные сетки и компоненты, чтобы интерфейс `GRACEIT` выглядел нормально и на desktop, и на mobile.

## 4. UI/UX принципы

### UI
UI (`User Interface`) — это визуальная часть интерфейса:
- цвета
- кнопки
- карточки
- шрифты
- иконки
- расположение блоков

### UX
UX (`User Experience`) — это опыт пользователя:
- насколько понятно пользоваться приложением
- насколько быстро можно выполнить действие
- удобно ли искать нужную информацию
- понятны ли ошибки и подсказки

### Основные принципы UI/UX
- простота: интерфейс не должен быть перегружен
- консистентность: одинаковые элементы должны вести себя одинаково
- предсказуемость: пользователь должен понимать, что произойдет после нажатия
- обратная связь: приложение должно показывать загрузку, ошибки, успешные действия
- доступность: интерфейс должен быть удобным для разных пользователей
- адаптивность: корректная работа на разных экранах

### Что в проекте
В `GRACEIT` использованы:
- темная тема
- единая система карточек и кнопок
- состояния `loading`, `error`, `empty`
- удобная навигация

## 5. SEO

SEO (`Search Engine Optimization`) — это оптимизация сайта для поисковых систем.

Цель SEO:
- чтобы сайт лучше индексировался
- чтобы страницы могли появляться выше в поиске

Для SEO важны:
- корректная HTML-структура
- заголовки
- мета-теги
- понятные URL
- быстрая загрузка
- доступность контента для поисковых роботов

### Почему у SPA есть нюансы с SEO
В SPA контент часто подгружается через JavaScript уже после загрузки страницы. Поисковые роботы не всегда корректно обрабатывают такие сайты.

Поэтому для SEO в сложных проектах часто используют:
- SSR (`Server-Side Rendering`)
- SSG (`Static Site Generation`)
- специальные SEO-мета-теги

### Что это значит для данного проекта
Для учебного или pet-project SPA может быть вполне нормальным решением. Но если продвигать проект в поиске, одной обычной SPA может быть недостаточно.

## 6. PWA

PWA (`Progressive Web App`) — это веб-приложение, которое по поведению становится похоже на нативное мобильное приложение.

Возможности PWA:
- установка на устройство
- работа через иконку на домашнем экране
- офлайн-режим или частичная работа без сети
- push-уведомления
- кэширование ресурсов

Для PWA обычно нужны:
- `manifest.json`
- `Service Worker`
- HTTPS

### Что с этим проектом
Сейчас `GRACEIT` — это обычное веб-приложение. Если добавить manifest и service worker, его можно развивать в сторону PWA.

## 7. Образы и контейнеры Docker

### Docker image
Docker image — это шаблон, по которому создается контейнер.

Проще говоря:
- image — это "подготовленная коробка"
- в ней уже описано, что должно быть установлено и как запускаться

### Docker container
Контейнер — это уже запущенный экземпляр image.

То есть:
- image — рецепт
- container — работающий результат

### Зачем это нужно
- приложение одинаково запускается на любой машине
- меньше проблем с окружением
- проще переносить проект между компьютерами и серверами

### Что в проекте
В проекте есть Docker-конфигурация для запуска:
- frontend
- backend
- PostgreSQL

## 8. Docker Compose

Docker Compose нужен для запуска нескольких контейнеров сразу.

Например, в одном проекте можно поднять:
- frontend
- backend
- базу данных

Одной командой:

```bash
docker compose up --build
```

### Что делает Compose в этом проекте
`docker-compose.yml` описывает:
- контейнер frontend
- контейнер backend
- контейнер PostgreSQL

То есть локально проект можно запускать как полноценную систему из нескольких сервисов.

## 9. CI/CD

CI/CD — это автоматизация сборки, тестирования и деплоя.

### CI
CI (`Continuous Integration`) — это непрерывная интеграция.

Смысл:
- разработчик пушит код
- система автоматически запускает сборку и тесты
- если что-то сломано, это сразу видно

### CD
CD (`Continuous Delivery` / `Continuous Deployment`) — это автоматическая доставка или выкладка приложения.

Смысл:
- после успешной сборки проект можно автоматически развернуть на хостинге
- после деплоя можно автоматически проверить, что приложение живо

### Что настроено в проекте
Через GitHub Actions настроены:
- backend build + test
- frontend build
- Docker build
- deploy backend
- deploy frontend
- backend healthcheck
- frontend healthcheck

## 10. SaaS, PaaS, IaaS, IaC

### SaaS
SaaS (`Software as a Service`) — это готовое программное обеспечение как сервис.

Примеры:
- Google Docs
- Notion
- Trello

Пользователь просто пользуется программой, не думая о серверах и инфраструктуре.

### PaaS
PaaS (`Platform as a Service`) — это платформа, на которой можно разместить свое приложение.

Примеры:
- Render
- Heroku
- Railway

Разработчик загружает код, а платформа сама помогает его запускать.

### IaaS
IaaS (`Infrastructure as a Service`) — это инфраструктура как сервис.

Примеры:
- AWS EC2
- DigitalOcean Droplets
- Google Compute Engine

Тут разработчику дают серверы, диски, сеть, а дальше он сам все настраивает.

### IaC
IaC (`Infrastructure as Code`) — это описание инфраструктуры через код.

Примеры инструментов:
- Terraform
- Ansible
- Pulumi

Смысл:
- инфраструктура настраивается не руками, а через файлы конфигурации
- это удобно для повторяемости и автоматизации

## 11. Что происходит при деплое приложения

Когда приложение деплоится, обычно происходит такая цепочка:

1. Разработчик отправляет код в репозиторий.
2. CI/CD запускает сборку проекта.
3. Выполняются тесты.
4. Собираются артефакты или Docker-образы.
5. Платформа разворачивает новую версию приложения.
6. Приложение запускается с нужными переменными окружения.
7. Выполняется healthcheck.
8. Если healthcheck успешный, новая версия считается рабочей.

### В контексте этого проекта
Для `GRACEIT` при деплое:
- GitHub Actions проверяет backend и frontend
- Render получает команду на деплой
- backend подключается к PostgreSQL
- frontend публикуется и начинает обращаться к backend
- healthcheck проверяет, что сервисы работают

## 13. Как объяснить pipeline на защите

### Что такое pipeline
Pipeline — это последовательность автоматических шагов, которые выполняются после изменения кода.

Проще говоря, pipeline отвечает на вопрос:

> Что должно автоматически произойти после того, как разработчик отправил код в репозиторий?

Обычно туда входят:
- сборка проекта
- тестирование
- деплой
- проверка, что приложение после деплоя работает

### Какой pipeline настроен в этом проекте
В проекте pipeline настроен через `GitHub Actions`.

После `push` в ветку `main` автоматически запускается workflow, который:

1. собирает и тестирует backend
2. собирает frontend
3. проверяет Docker-сборку backend
4. деплоит backend на Render
5. деплоит frontend на Render
6. выполняет healthcheck backend
7. выполняет healthcheck frontend

### Как читать pipeline в GitHub Actions
В GitHub Actions pipeline отображается не как одна линия, а как набор job-ов.

Это значит:
- некоторые этапы могут идти параллельно
- некоторые этапы ждут завершения других

Например:
- `Backend Build & Test` и `Frontend Build` могут выполняться независимо
- `Docker Build` зависит от успешной сборки
- `Deploy` запускается только после успешных проверок
- `Healthcheck` идет после деплоя

### Что означают job-ы в этом проекте

#### Backend Build & Test
Проверяет backend:
- устанавливает Java
- запускает `mvn verify`
- выполняет сборку, тесты и проверки стиля кода

#### Frontend Build
Проверяет frontend:
- устанавливает Node.js
- запускает `npm ci`
- запускает `npm run build`

#### Docker Build
Проверяет, что backend Docker-образ успешно собирается.

#### Deploy Backend to Render
Отправляет backend на Render через deploy hook.

#### Deploy Frontend to Render
Отправляет frontend на Render через deploy hook.

#### Backend Healthcheck
После деплоя проверяет, что backend доступен и отвечает через:

```text
/actuator/health/readiness
```

#### Frontend Healthcheck
После деплоя проверяет, что frontend URL открывается.

### Как это можно объяснить простыми словами
Если нужно объяснить коротко и понятно, можно сказать так:

> После отправки кода в GitHub автоматически запускается pipeline.  
> Сначала система проверяет backend и frontend: собирает проект, запускает тесты и проверяет Docker-сборку.  
> Если все успешно, GitHub Actions отправляет команды на деплой backend и frontend на Render.  
> После деплоя выполняется healthcheck, чтобы убедиться, что сервисы реально работают.

### Если преподаватель спросит, почему pipeline выглядит не как одна линия
Можно ответить так:

> GitHub Actions показывает pipeline как граф зависимостей.  
> Поэтому часть задач выполняется параллельно, а часть ждет завершения предыдущих этапов.  
> Логика при этом остается стандартной: сборка, тестирование, деплой и healthcheck.

### Короткая версия для ответа

> В проекте настроен CI/CD pipeline через GitHub Actions.  
> После push автоматически выполняются сборка backend и frontend, тестирование backend, проверка Docker-сборки, деплой на Render и healthcheck обоих сервисов.
